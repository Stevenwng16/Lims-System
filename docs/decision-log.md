# Decision log

One line per fundamental implementation choice, **at the moment it is made** — anything that realises an ADR, shapes the data model, or a future developer would ask "why is this like this?" about. Story-level details don't belong here. This log feeds the validation package for accredited customers.

| 3 Jul 2026 | US-B3 **availability is computed on every read, never stored** — `equipmentAvailability()` derives Blocked/Due soon/Available from calibration due date, active check types' latest entries and the out-of-service flag; calendar-day comparisons (yyyy-mm-dd strings), consistent with QC expiry | AC 6/7 literally; a stored state column would be exactly the thing a "manual unblock" could flip (hard-never list) | US-B3 AC 6/7 |

| 3 Jul 2026 | US-B3 **a scheduled (daily/weekly) check that was never performed counts as overdue → Blocked**; per-use checks never block on schedule (they gate at use, epic D) — only on a failed last result | fitness for use must be *proven*; "no evidence yet" defaulting to Available would let a fresh check requirement be ignored | US-B3 AC 6 / §6.4 |

| 3 Jul 2026 | US-D1 decision block (Ramazan): per-(sample × method) progress is **derived, never stored** — computed live from open-batch membership, batch step position and (from US-D6) completion records; the stored `sample.status` column is removed and every C2/C3 read goes through the derived view | a stored status is a copy of reality that every future event (set-back, void, completion) must remember to update — the drift-bug class a traceability product cannot afford; consistent with ADR-2 ("current result is a view") and B3's computed availability. Chosen over an explicit state table | US-D1 AC 4 / ADR-2 |
| 3 Jul 2026 | US-D3 decision block (Ramazan): step workflow = **minimal state + append-only transition events** (ADR-2 pattern, the story's advice) — the batch stores only `currentStepIndex` + phase; every advance (with its structured equipment selection), set-back (with reason) and void is appended to the batch's existing event log; the Steps rail and the History tab are two renderings of that one list | one mutable fact + immutable history satisfies all three AC constraints directly: redo = new event (originals retained), History is a pure view, and concurrency = compare expected vs stored step before writing; chosen over a separate completion table with replayed position | US-D3 AC 2/6/10 / ADR-2 |
| 4 Jul 2026 | US-D4 decision block 1 (Ramazan): measurement records = **append-only rows + validity status column** (ADR-2 open question 3 closed with ADR-2's own proposal) — one row per saved value carrying links (batch · sample/QC · analyte · method version), value, origin, attribution, a `supersedes` pointer + mandatory reason; the current cell value is the newest record (pure view); `validity` (pending → valid/rejected) is flipped only by US-D6 review, each flip an audit event | same shape as the D1/D3 choices: minimal mutable state, immutable history, no replay machinery; chosen over full event sourcing | US-D4 AC 7/8 / ADR-2 |
| 4 Jul 2026 | US-D4 decision block 2 (Ramazan): numeric results stored as the **canonical decimal string exactly as entered** (full precision; "0.010" never collapses to 0.01 — entered precision is part of the original observation, ALCOA); comparisons run on exact BigInt arithmetic (as in B3); the real backend keeps the string as the original and may add a decimal shadow column for querying — floats never exist anywhere | trailing zeros are significant in a lab record; this is the CLAUDE.md hard rule applied to its most important case; chosen over a normalizing decimal column | US-D4 AC 5/6 / ADR-4 |
| 4 Jul 2026 | US-D4 **ambiguity rule for locale-aware numeric entry** (ADR-4 "never guess"): a single comma OR point is accepted as the decimal separator, EXCEPT the classic thousands pattern — exactly 3 digits after the separator with a 1–3-digit non-zero integer part (e.g. "1,234" / "12.345") — which is rejected with a message to disambiguate ("1234" or "1,2340"); mixed/repeated separators always rejected. "0,010"-style values (zero integer part) are never ambiguous and pass | a Dutch analyst's "1.234" plausibly means 1234; symmetric treatment of both separators is the only honest reading of "locale-aware, never guessed" | US-D4 AC 5 / ADR-4 |
| 4 Jul 2026 | US-D4 **bulk paste and worksheet auto-read never overwrite silently**: occupied cells are skipped in both flows (marked "occupied — correct manually with a reason"); worksheet confirm re-reads and re-validates the file server-side, so origin `worksheet` can never be forged onto hand-typed values | a blanket paste-over would launder supersedes past the mandatory per-correction reason (AC 8); provenance must be provable, not claimed | US-D4 AC 8/13/14 |
| 4 Jul 2026 | US-D5 decision block 1 (Ramazan): import parsing = **own strict CSV parser + exceljs for Excel, raw text out of both** — CSV with RFC 4180 quoting and the delimiter DECLARED in the configuration (never sniffed, like the decimal separator); .xlsx read server-side via exceljs (MIT, npm, our first runtime dependency) extracting cell TEXT so the library never interprets numbers; every value then passes our declared-separator ADR-4 validation | ADR-4's reject-never-guess must stay the only judge of values; SheetJS-style all-in-one parsing coerces cells to floats/dates on read — the exact behaviour the hard-never list forbids | US-D5 AC 1/5 / ADR-4 |
| 4 Jul 2026 | US-D5 decision block 2 (Ramazan): the import event stores a **self-contained JSON snapshot** — original file + SHA-256, the mapping frozen exactly as applied, actor, timestamp, per-row outcomes — appended to the batch; configs stay simple audited masterdata (no versioning) | AC 8's "reproducible from the event alone" answered by ONE record, same shape as working copy / worksheet versions; chosen over normalized event tables | US-D5 AC 8 / ADR-3 |
| 4 Jul 2026 | US-D5 **duplicate rows for the same (QC entry × analyte) within one import**: the first row imports, later duplicates are rejected per cell with the reason shown — all rows still attach to the entry for the count-vs-quantity notice | the MVP result model has one cell per (entry × analyte); QC position bookkeeping is deliberately parked (scope note §6) — silently overwriting the first measurement would lose an original observation | US-D5 AC 4 / US-D1 AC 7 |
