# Decision log

One line per fundamental implementation choice, **at the moment it is made** — anything that realises an ADR, shapes the data model, or a future developer would ask "why is this like this?" about. Story-level details don't belong here. This log feeds the validation package for accredited customers.

| Date | Decision | Why | Relates to |
|---|---|---|---|
| 3 Jul 2026 | Auth provider: **Supabase Auth (Pro plan)**, with every auth call proxied through our own Next.js server routes (no direct browser→Supabase auth calls) | server-proxy gives synchronous capture of every auth event incl. failed logins into our append-only audit log (invariant 1, US-A1 AC 3/9); per-org password policy, lockout counter and MFA requirement enforced server-side directly from our org settings (AC 4/5/7, invariant 7); TOTP included, lowest cost ($25/mo). Chosen over Auth0 (vendor-native AC coverage but $150+/mo, settings-sync layer, ~90-org connection cap) — full analysis in `docs/research/us-a1-auth-provider-options.md`. Accepted trade-offs: lockout/MFA-enforcement logic is our code to test & defend; enterprise SSO is SAML-only | US-A1 / ADR-1 |
| 3 Jul 2026 | App database: **Supabase Postgres (EU region) for everything** — identity and app data in one database; tenant isolation via shared DB + `organisation_id` on every table + Postgres row-level security | one DB makes invariant 5 enforceable and testable in a single place with RLS tied to the auth context; lowest ops load for a one-dev team; dissolves the two-database split that Supabase-Auth-plus-Azure would create. Departs from the non-binding Azure-stack advice — its main payoff (Entra) fell away in the US-A1 auth research | ADR-1 / US-A1 / US-A2 |
| 3 Jul 2026 | Frontend built against a **mock API layer** behind an internal interface (Next.js route handlers + in-memory store) until the partner-built backend's spec is known; real backend plugs in as a thin adapter | dev-partner backend exists but stack/auth provider still unknown; this keeps story work unblocked while keeping the swap surface small. Note: supersedes nothing yet — the two Supabase entries above stand until the partner's stack is confirmed | US-A1+ (all frontend) |
| 3 Jul 2026 | UI foundation: **shadcn/ui** on Tailwind v4 (components vendored into the repo), with react-hook-form + zod for forms | accessible, professional components for a form/table-heavy LIMS without a runtime library lock-in; vendored code fits a validated product | all frontend stories |
| 3 Jul 2026 | Rendering model (US-A3 decision block): **server-rendered Next.js App Router** with client-component islands for interactivity — not a SPA | server-side rendering keeps every permission/data decision on the server (invariant 4), gives fast first paint on lab terminals, and was implicit in the Next.js scaffold choice; client islands (forms, dialogs, sidebar) give SPA-feel where it matters | US-A3 / all frontend |
| 4 Jul 2026 | ADR-3 file-storage mechanism (US-B1 decision block): **formally deferred** until the partner backend's stack is confirmed — same status as the Supabase entries. The mock attachment facility already honours ADR-3's requirements (immutable versions, SHA-256 checksum computed from real bytes, tenant scoping), so no frontend rework follows from any later mechanism choice | Ramazan's call; deciding now (e.g. Azure Blob per the non-binding advice) could be invalidated by the unknown backend | ADR-3 / US-B1 |
| 4 Jul 2026 | **Lab managers may reset/unlock** their own analyst & read-only users (US-A6 reading), scoped to shared labs — server-enforced | US-A6 grants lab managers full management of their labs' analyst/read-only users; chosen over US-A1's "reset/unlock = Admin-only". Ramazan's call | US-A1 / US-A6 (epic-A audit) |
| 4 Jul 2026 | **Master-data lists are exempt from active-lab scoping** — the header lab switcher sets the active *working context* (where new jobs/batches are created), but reference lists (Methods, later QC/Equipment) show all of a user's labs (US-B1 AC 1) | a multi-lab manager wants to see all their methods at once; matches US-B1's cross-lab UI sketch. ⚠ **Needs a Notion amendment to US-A3 AC 4**: "all screens show data of the active lab only" → "all work screens" | US-A3 AC 4 / US-B1 AC 1 (epic-A audit) |
| 4 Jul 2026 | **Vendor support sessions show an "All labs" shell indicator**, no lab switcher — support data is org-wide (per the 3 Jul routing decision) | consistent with org-wide support visibility; a per-lab vendor switcher adds machinery for no benefit | US-A3 AC 4 / US-A2 (epic-A audit) |
| 4 Jul 2026 | **Settings keeps its six-card layout** rather than three literal AC-9 sections | all AC-9 controls are present and functional; CLAUDE.md treats UI sketches as indicative | US-A7 AC 9 (epic-A audit) |
| 4 Jul 2026 | **Deactivation requires a reason** on labs and methods (mandatory field, mirroring org suspension) — invariant 2 says "deactivated or voided with a reason" even though US-A5/US-B1 ACs don't ask for it; chosen over logging an accepted deviation | audit finding; the invariant text is explicit and reasons are exactly what an ISO 17025 auditor asks for | invariant 2 / US-A5 / US-B1 |
| 3 Jul 2026 | Org routing: **session-bound, no organisation in the URL** (clean paths like `/jobs/J-2026-0142`; the authenticated session carries the organisation context) | every user belongs to exactly one organisation (US-A1 AC 13) so the org is implicit; URLs can never leak or address another tenant; simplest to build and test; org slugs stay reserved so path- or subdomain-based routing can be layered on later if enterprise SSO/branding demands it | ADR-1 / US-A2 |
